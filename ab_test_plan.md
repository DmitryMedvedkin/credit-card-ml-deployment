# A/B тестирование моделей прогнозирование дефолта

## 1. Цель и гипотеза

**Цель:** проверить, улучшает ли переход с текущей модели v1  на новую v2 качество прогнозирования дефолта по кредитным картам на реальном трафике.

**Модели:**
- **Контроль (v1)** - LogisticRegression, текущая модель
- **Тест (v2)** - RandomForestClassifier, новый кандидат

### Двухуровневая постановка гипотез

Решение о внедрении принимается по бизнес-эффекту, а техническая метрика служит быстрым индикатором в ходе теста.

**Бизнес-гипотеза (решающая):**
- H0: модель v2 не снижает ожидаемые финансовые потери по сравнению с v1;
- H1: модель v2 снижает ожидаемые потери от невозвратов при сохранении или росте доли одобренных заявок.

**Техническая гипотеза:**
- H0 (нулевая): новая модель не отличается от текущей по основной метрике (F1_v2 − F1_v1 = 0);
- H1 (альтернативная): новая модель повышает основную метрику (F1_v2 − F1_v1 > 0).

Офлайн-метрики на тестовой выборке уже показывают преимущество модели v2 (F1 = 0.53 против 0.47, ROC-AUC 0.77 против 0.72). A/B-тест проверит, сохраняется ли это преимущество на живом потоке заявок, где распределение может отличаться от обучающего.

## 2. Дизайн эксперимента

### 2.1 Разделение трафика

Каждая входящая заявка случайно направляется на одну из моделей с вероятностью 50%. При отсутствии явного `model_version` версия выбирается через `random.choice(["v1", "v2"])`, а поле `routing` фиксируется как `ab_random`.
Случайная рандомизация гарантирует, что обе группы статистически однородны по составу клиентов (возраст, лимит, история платежей). Любое систематическое деление внесло бы смещение, разница в метриках могла бы объясняться различием групп, а не модели. При случайном делении единственное различие между группами - это сама модель.

### 2.2 Продолжительность теста

Тест длится до накопления достаточной выборки в каждой группе, но не менее полного бизнес-цикла, чтобы покрыть недельную сезонность. Необходимый минимальный размер выборки оценен заранее по формуле для сравнения двух пропорций:
n = (z_(a/2) + z_B)^2 * (p1(1−p1) + p2(1−p2)) / (p1 − p2)^2
где:
- a = 0.05 (уровень значимости)
- z_(a/2) = 1.96
- B = 0.2 (мощность = 80%), z_B = 0.84
- p1, p2 - ожидаемые значения метрик в группах
- (p1 - p2) - минимальный значимый эффект

n = (1.96 + 0.84)^2 * (0.63 * 0.37 + 0.56 * 0.44) / 0.07^2 = 770

Для наших моделей, где офлайн Recall v1 = 0.63, v2 = 0.56, разница 7 п.п., при α = 0.05 и мощности 80% требуется около 770 наблюдений с известным исходом в каждой группе (через statsmodel - 771). Однако, это необходимое число, на пракктике может потребоваться больше наблюдений.

## 3. Метрики

### 3.1 Основная метрика

**F1-score для класса дефолт**
Выбрана как основная, потому что класс дефолта несбалансирован, accuracy будет обманчива. F1 одновременно штрафует и за пропуск дефолта, и за ложные срабатывания.

### 3.2 Дополнительная метрика

**Recall для классу дефолт**
В кредитном скоринге ошибки несимметричны по цене. Пропуск дефолта, выдать кредит тому, кто не вернёт, ведёт к прямому финансовому убытку в размере невозврата. Ложная тревога False Positive - отказать надёжному клиенту ведёт к упущенной прибыли, которая обычно меньше суммы невозврата. Поэтому для банка важнее не пропускать дефолтеров, т.е. максимизировать Recall

### 3.3 Бизнес-метрики

В качестве отслеживаемых бизнес показателей выбираем:

- **Ожидаемые финансовые потери:**
 Сумма потерь от невозвратов пропущенных дефолтеров и упущенной прибыли из-за отказа хорошим клиентам. Снижение ожидаемых финансовых потерь - главный аргумент.

- **Доля одобренных заявок при фиксированном уровне риска:**
 Если модель v2 точнее отсеивает плохих клиентов, то среди одобренных будет столько же дефолтов, сколько и раньше, но самих одобрений можно сделать больше - рост выручки без роста риска.

## 4. Статистический анализ

### 4.1 Тест для пропорций (Recall, Precision)

Recall и Precision - доли (пропорции), а признак обнаружили дефолтера / не обнаружили - бинарная величина с распределением Бернулли. Для сравнения двух независимых групп применяется z-критерий для пропорций - стандартный и наиболее распространённый тест для оценки значимости разницы долей.

```python
from statsmodels.stats.proportion import proportions_ztest

# count - число «успехов» (пойманных дефолтеров) в каждой группе
# nobs - число наблюдений (реальных дефолтеров) в каждой группе
stat, p_value = proportions_ztest(
    count=[tp_v1, tp_v2],
    nobs=[pos_v1, pos_v2],
    alternative='larger' # Правосторонняя: проверяем, что v2 лучше v1
)
```
### 4.2 Критерий принятия решения по p-value

- если p-value <= a (0.05) - результат статистически значим, отвергаем H0 в пользу H1 (новая модель значимо лучше);
- если p-value > a - значимой разницы не обнаружено, не отвергаем H0.

В спорных ситуациях (p-value близко к a) предпочтение отдаётся текущей модели v1, так как её сохранение не требует затрат на внедрение.

### 4.3 Доверительный интервал разницы пропорций

Дополнительно к тесту строим 95% доверительный интервал разницы через diff_proportion_conf_interval:

```python
def diff_proportion_conf_interval(x_p, n, gamma=0.95):
    alpha = 1 - gamma # уровень значимости
    diff = x_p[1] - x_p[0] # выборочная разница конверсий групп B и A
    z_crit = -norm.ppf(alpha/2) #z-критическое
    eps = z_crit * (x_p[0] * (1 - x_p[0])/n[0] + x_p[1] * (1 - x_p[1])/n[1]) ** 0.5 # погрешность
    lower_bound = diff - eps # левая (нижняя) граница
    upper_bound = diff + eps # правая (верхняя) граница
    # возвращаем кортеж из  границ интервала
    return lower_bound, upper_bound
```

(p_v2 − p_v1) +- z_(крит) * (p_v1(1−p_v1)/n1 + p_v2(1−p_v2)/n2)^(1/2)

Интерпретация по трём случаям:
- обе границы > 0 - v2 значимо лучше v1
- обе границы < 0 - v1 значимо лучше v2
- интервал охватывает 0 - значимой разницы нет

### 4.4 Бутстрап F1

Для основной метрики F1 будем использовать бутсрап:
1. Из каждой группы 10000 раз формируем подвыборку с возвращением
2. На каждой считаем  F1
3. По всем посчитанным разницам, F1_v2 − F1_v1, строим 95% доверительный интервал (2.5-й и 97.5-й перцентили)

Если весь 95% интервал лежит выше нуля (обе границы положительны), значит F1 у v2 действительно выше, чем у v1, соответственно улучшение статистически значимо. Если интервал захватывает ноль - разница может быть случайной, значимого улучшения нет.

### 4.5 Критерий успешности теста

v2 признается победителем и выкатывается в продакшен на 100% трафика, если одновременно выполняются три условия:
1. Бутстрап подтверждает значимое улучшение F1
2. Recall v2 не ниже чем recall v1 или его снижение компенсируется ростом precision и снижением бизнес-потерь.
3. Ожидаемые финансовые потери у модели v2 ниже, чем у v1

Если знгачимого улучшения нет, оставляем v1

## 5. Связь с архитектурой

- **Разделение трафика** обеспечивает A/B-роутинг в `api.py` (случайный выбор версии, поле `routing: ab_random`). При анализе берутся только записи с `routing: ab_random` - ручные проверки (`explicit`)исключаются, чтобы не искажать статистику.
- **Сбор данных** обеспечивает JSON-логирование: на каждый вызов пишется запись с признаками клиента, выданной вероятностью, прогнозом и версией модели (`model_version`). Такие логи собираются, что позволяет считать метрики отдельно по каждой группе.
- **Версионирование** (артефакты с полем `version`) гарантирует, что в логах видно, какая именно модель выдала каждый прогноз.
